# Manufacturing Digital Solution Framework

Use this reference when the deck is about manufacturing digital transformation, smart factory, MOM/MES/WMS/QMS/TPM/APS, Industry 4.0, or AI for manufacturing operations.

Core rule: **solve the business problem before designing slides**. Build a concise internal `Solution Brief`, then turn it into a PPTX-ready deck.

## 1. Solution Spine

Use this sequence for most manufacturing digital solution decks:

1. **Business thesis**: why the company must change now.
2. **Enterprise diagnosis**: what is known or assumed about business, process, systems, data, organization, and IT/OT.
3. **Pain and root cause**: what symptoms are visible and what causes them.
4. **Value opportunities**: which operational or financial outcomes are worth pursuing.
5. **Target blueprint**: target business, capability, system, data, AI, and operating model.
6. **Priority scenarios**: concrete closed loops that prove the transformation.
7. **Architecture**: how systems, data, and shop-floor assets interact.
8. **Roadmap**: phases, acceptance criteria, rollout scope.
9. **Governance**: business owner, data owner, KPI owner, PMO, change management.
10. **Value proof**: KPI baseline, target, assumption, and confidence.

## 2. Internal Solution Brief

Before creating slides, draft this brief internally:

| Field | Required content |
|---|---|
| Industry and model | discrete/process/mixed, MTO/MTS/ATO/ETO, single/multi-plant |
| Business thesis | growth, cost, delivery, quality, flexibility, compliance, export, or service model pressure |
| Current-state assumptions | business, process, system, data, organization, IT/OT facts or assumptions |
| Pain and root cause | separate symptom, pain, root cause, business impact |
| Value opportunities | reduce cost, improve delivery, improve quality, improve asset use, reduce risk, support growth |
| Priority scenarios | 2-5 scenario closed loops with trigger, systems, data, KPI |
| Target blueprint | business capability, system role, data flow, AI use, operating mechanism |
| Roadmap | pilot, core build, scale, continuous operation |
| KPI confidence | exact/range/directional, with source or assumption |
| Open questions | information needed before customer-ready precision |

Do not expose this whole table in the final deck unless the user asks for working notes.

## 3. Deck Types

### Sales / Presales Solution Deck

Best when the user wants a client-facing proposal.

Recommended flow:

1. Cover
2. Business thesis
3. Current-state diagnosis
4. Pain and root-cause map
5. Value opportunity map
6. Target business capability blueprint
7. Priority scenario closed loops
8. System/data/AI architecture
9. Implementation roadmap
10. KPI and value realization
11. Governance and next step

### Executive Transformation Deck

Best for GM/CEO/CIO/COO discussion.

Recommended flow:

1. Transformation thesis
2. Maturity baseline and strategic gap
3. Value pool by domain
4. Target operating model
5. Multi-year roadmap
6. Investment, KPI, and governance
7. Decisions requested

### POC / Pilot Proposal

Best when the goal is to start small and prove value.

Recommended flow:

1. Pilot objective and boundary
2. Current pain and baseline assumptions
3. Target scenario closed loop
4. Pilot architecture and data scope
5. 6-12 week workplan
6. Acceptance criteria and KPI
7. Scale path after pilot

### Product-to-Solution Conversion

Best when the user provides product functions and asks for a solution deck.

Rules:

- Convert features into customer scenarios.
- Convert modules into capability boundaries.
- Convert screenshots into optional proof, not the main argument.
- Convert product benefits into KPI assumptions and operational outcomes.

## 4. Slide Mapping

| Solution element | Best slide pattern |
|---|---|
| Business thesis | Executive Summary / Gap Analysis |
| Enterprise diagnosis | Diagnosis / Maturity Heatmap |
| Pain and root cause | Root Cause / Issue Tree |
| Value opportunities | Benefits / KPI Dashboard / Value Map |
| Target blueprint | Architecture / Capability Boundary |
| Priority scenarios | Scenario Closed Loop / Process Pathway |
| System and data architecture | Architecture / Data Flow |
| Roadmap | Timeline / Action Board |
| Governance | Action Board / RACI-style table |
| KPI proof | KPI Dashboard / KPI Tree |

## 5. Quality Rules

- Do not start with a system list unless the user explicitly asks for a technical architecture deck.
- Every pain point must connect to a business object: order, work order, lot, inventory, inspection, equipment, shipment, cost, cash, or customer complaint.
- Every target capability should name the operational mechanism it changes.
- AI content must name the scenario, data prerequisite, decision support, and human approval boundary.
- When data is missing, write assumptions and use ranges or directional wording.
