# Consulting Frameworks

Use these frameworks to turn manufacturing solution content into a clear deck. Keep the final wording client-facing; do not expose internal reasoning unless useful.

## SCR / SCQA

Use for the overall deck:

- **Situation**: shared business context and current reality.
- **Complication**: pain, gap, risk, or missed value.
- **Resolution**: proposed solution, architecture, roadmap.
- **Evidence**: KPI, case, baseline/target, governance, next step.

Title sequence test: read only slide titles; they should tell the story.

## MECE

Use for diagnosis, scope, and value breakdown.

Rules:

- categories should not overlap
- coverage should be complete for the chosen scope
- same level of granularity
- 3-7 categories preferred

Manufacturing examples:

- Plan / Source / Make / Deliver / Return
- Production / Quality / Warehouse / Equipment / Planning
- Process / System / Data / Organization / IT-OT

## Pyramid Principle

Use for executive summaries and decision pages:

1. conclusion first
2. 2-4 supporting arguments
3. evidence or assumption row

Every action title should be a complete conclusion, not a topic label.

## Issue Tree

Use for root cause and scope definition.

Example:

```text
交付不稳定
├─ 计划频繁变更
├─ 物料齐套不可见
├─ 产线进度滞后
└─ 质量异常处理慢
```

Each branch should become either a scenario, a data requirement, or a governance action.

## 2x2 Matrix

Use for priority selection:

- impact vs feasibility
- value vs complexity
- urgency vs readiness
- standardization vs local flexibility

Preferred for pilot selection and roadmap prioritization.

## Value Chain

Use when the deck needs an end-to-end operation view.

Manufacturing chain:

```text
Plan -> Source -> Make -> Quality -> Store -> Deliver -> Service
```

Use support bands for data, system integration, and governance.

## Before / After

Use for current vs target operating model.

Keep each side to 3-5 claims. The "after" side must name the operational mechanism, not just an aspiration.

## Waterfall / Bridge

Use for KPI or financial value logic:

- baseline
- improvement levers
- target
- assumptions

If data is weak, use ranges and assumption notes.

## Consulting Voice

Avoid:

- "赋能", "打造闭环", "构建生态" without a concrete object
- "标志着", "里程碑", "新纪元"
- fake authorities such as "业内认为"
- slogan endings

Prefer:

- concrete business objects
- short claims
- measurable outcome
- explicit assumption

Example:

- Weak: `通过数据驱动赋能制造运营闭环。`
- Better: `把计划变更同步到产线、仓储和质检，先减少插单和缺料导致的停线。`
